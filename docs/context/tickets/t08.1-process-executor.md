# T08.1: Project Correction — ProcessExecutor infrastructure injection

- **type**: tech-debt / cross-cutting
- **complexity**: medium
- **layer**: `infrastructure/` + `generation/` + `plugins/`
- **dependencies**: T-006, T-007
- **phase**: 1 — MVP Foundation
- **estimated_context**: ~15% of window

## Description

Architecture Rule #4 states that *"Infrastructure is the only I/O layer — no filesystem writes or subprocess calls outside `infrastructure/`."* However, `CommandRunner.generate()` (in the `plugins/` layer) needs to run subprocess calls (e.g. `uv add`) but has no infrastructure mechanism to do so. The `PluginExecutionEngine` calls `plugin.generate(spec, output_dir)` directly without injecting any I/O wrapper.

Create the missing `ProcessExecutor` class in `infrastructure/` and thread it through the pipeline so plugins can run subprocess calls without violating the I/O layer rule.

## Changes

### 1. Create `src/forge/infrastructure/process_executor.py`

```python
class ProcessExecutor:
    def run(self, cmd: list[str], cwd: Path | None = None) -> None: ...
```

Wraps `subprocess.check_call`. The abstraction lives in `infrastructure/` — tests mock the executor directly instead of patching `subprocess`.

### 2. Update `CommandRunner` interface (`plugins/base.py`)

Add `executor: ProcessExecutor` parameter to `generate()`:

```python
class CommandRunner(ABC):
    @abstractmethod
    def generate(self, spec: ProjectSpec, target_dir: Path, executor: ProcessExecutor) -> None: ...
```

### 3. Update `PluginExecutionEngine` (`generation/stages/plugin_execution_engine.py`)

- Accept `executor: ProcessExecutor` in `__init__`
- Pass `self._executor` to `plugin.generate(spec, output_dir, self._executor)`

### 4. Update `Orchestrator` (`generation/orchestrator.py`)

- Accept optional `executor: ProcessExecutor | None = None` in `__init__` (defaults to creating one)
- Pass executor to `PluginExecutionEngine` in the default stages list

### 5. Update all test mock plugins

`MockCommandPlugin`, `CommandOnlyPlugin`, `FullPlugin`, and the `patched_generate` in checkpoint tests must accept the new `executor` parameter (even if unused).

### 6. Fix AC-4 scanner (`tests/unit/test_plugin_base.py`)

Change `plugins_dir.glob("*.py")` → `plugins_dir.rglob("*.py")` so subdirectory plugins like `fastapi/plugin.py` are scanned for cross-layer import violations.

## Test plan

- Existing PluginExecutionEngine tests continue to pass: all mock `generate()` accept the new parameter without using it
- New `ProcessExecutor.run()` test verifies it delegates to `subprocess.check_call`
- The `Orchestrator` default stages include the executor injection
- AC-4 scanner now discovers files in plugin subdirectories

## Acceptance Criteria

1. **Given** a `ProcessExecutor`, **when** `run(["echo", "hi"])` is called, **then** it executes the command via subprocess.
2. **Given** a plugin implementing `CommandRunner`, **when** `PluginExecutionEngine.run()` is called, **then** `generate()` receives the executor as the third argument.
3. **Given** the `Orchestrator` with default stages, **when** created, **then** `PluginExecutionEngine` receives a `ProcessExecutor`.
4. **Given** a `CommandOnlyPlugin` or `FullPlugin`, **when** `generate()` is called with an executor, **then** no error is raised (backward compatible).
5. **Given** the AC-4 scanner, **when** scanning `plugins/` directory, **then** files in subdirectories (e.g. `plugins/fastapi/plugin.py`) are included.
