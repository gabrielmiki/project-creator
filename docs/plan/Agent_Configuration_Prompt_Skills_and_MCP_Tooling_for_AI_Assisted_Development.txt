Agent conﬁguration, prompt skills, and MCP tooling for AI-assisted
development
The most impactful investment is not building more agents — it is engineering their context.
Across all three research areas, the dominant ﬁnding is that context quality determines agent
eﬀectiveness far more than prompt sophistication. Frontier LLMs can follow roughly 150–200
instructions before quality degrades uniformly, and Claude Code's system prompt already
consumes ~50 of those slots.  Every line in a system prompt, every MCP tool
deﬁnition, and every continuation document competes for a ﬁnite cognitive budget. The
research yields a clear priority order: conﬁgure the continuation prompt skill ﬁrst (it compounds
across all sessions), then the pre-commit and security diagnosis agents (they prevent the
highest-cost defects), then the remaining agents and skills incrementally. MCP tool deﬁnitions
alone can consume 72% of a 200K context window if loaded indiscriminately — making
dynamic toolset loading and the code-execution pattern essential, not optional.
The evidence base is uneven. Security review agents and MCP integration patterns are well-
documented by production teams. Architecture review and performance diagnosis agents have
emerging tooling but limited production benchmarks. TDD review agents at the planning gate
represent the largest evidence gap — no production-grade standalone tool exists. Where
consensus is absent, this report states so explicitly and provides the strongest available evidence.
Research Area 1: Specialist review agents
Architecture review works best on full plans with structured output
Top recommendation: Invoke the Architecture Review Agent at the planning gate on the
complete implementation plan (not per-ticket), requiring as input the ERD, module dependency
map, and any existing ADRs. Structure output as severity-bucketed ﬁndings
(Critical/High/Medium/Low) with speciﬁc component references and actionable remediation,
not narrative commentary.
The strongest production implementation is the Microsoft Architecture Review Agent (open-
sourced February 2026), which ingests architectural descriptions in any format and produces
prioritized risk analysis with component dependency mapping including fan-in/fan-out metrics
and orphaned component detection.  ArchGuard's Co-mate project from
the Thoughtworks open-source community takes the complementary approach — maintaining a
persistent architectural model and checking changes incrementally against it.  A third
pattern, the ArchGuard CLI, validates code changes against ADRs using semantic similarity plus
LLM reasoning, outputting speciﬁc violations with line references. 
The full-plan vs. per-ticket trade-oﬀ is real but resolvable. Full-plan review catches systemic
issues (single points of failure, coupling anti-patterns, missing components) that per-ticket
review misses entirely.  Per-ticket review is cheaper and faster but blind to

System prompt structure (targeting 150–250 words): role deﬁnition → context speciﬁcation

Security review requires deterministic SAST as input, not replacement
Top recommendation: Structure the Security Review Agent as a triage layer over deterministic
static analysis, not as a standalone reviewer. Feed it Bandit/pip-audit/Semgrep output alongside
the endpoint plan. Map OWASP Top 10 categories directly into the system prompt as evaluation
criteria. Invoke on every PR touching auth, payments, or external APIs.
The evidence here is the strongest of any agent category. Semgrep Assistant achieves 96%
agreement with human security researchers on true positive identiﬁcation  by
combining deterministic SAST with LLM reasoning.  By contrast, pure LLM security
review produced 88% false positives when applied to IDOR detection without SAST grounding
(Semgrep's own benchmark).  The hybrid pattern — deterministic tools ﬁnd candidates,
LLM triages and explains — is now industry consensus.
The minimum viable context set (derived from GitHub Security Lab's taskﬂow architecture)
consists of ﬁve elements: component inventory with entry points and trust boundaries, the
changed code/diﬀ with surrounding context, data ﬂow information (sources to sinks), the
dependency manifest, and previous triage decisions. Semgrep's "Assistant Memories" feature —
storing previous triage decisions for reuse — is a pattern worth replicating. 
For OWASP mapping, the system prompt should enumerate the categories the agent can
reliably detect. LLMs are strongest on injection (A03), broken access control (A01 in simple
cases), and cryptographic failures (A02). They struggle with security misconﬁguration (A05)
and SSRF (A10), which require architectural context.  The OWASP LLM Top 10 applies
when the system being reviewed includes LLM components — tools like Promptfoo implement
these as conﬁgurable test presets. 
Anti-pattern: Running LLM security review in CI/CD pipelines without input sanitization.
Aikido Security's "PromptPwnd" research found that untrusted user input injected into LLM
prompts in GitHub Actions can lead to credential theft, aﬀecting at least 5 Fortune 500
companies.  The review agent itself becomes an attack surface.

TDD review has the weakest tooling but the clearest evaluation criteria
Top recommendation: Build the TDD Review Agent to validate acceptance criteria against a
testability checklist derived from BABOK quality characteristics, operating on story ﬁles with
optional conftest context. The agent should ﬂag ambiguity patterns rather than generating test
code.
This is the least mature agent category. No production-grade, widely-adopted tool exists
speciﬁcally for pre-implementation test plan review. The closest implementations are a
Requirements Veriﬁcation Megaprompt pattern, the Meridian RFC (a GitLab agent validating
merge requests against acceptance criteria), and academic work from ICEIS 2025 on acceptance
criteria validation using NLP.
The testability checklist is well-established even without dedicated tooling. Testable criteria use
Given/When/Then structure with concrete preconditions and measurable outcomes. 
 Common ambiguity patterns that the agent should ﬂag include: vague qualiﬁers
("make it fast"), missing error scenarios, no measurable outcomes ("user is satisﬁed"),
implementation-coupled criteria ("use Redis"), and missing preconditions. The TDAD paper
(2026) found that a vanilla coding agent caused 562 pass-to-pass test failures across 100
instances  — validating the test plan before implementation is high-leverage.
Story ﬁles alone suﬃce for syntax validation, completeness checking, and ambiguity detection.
Architecture/conftest context becomes necessary for validating that test scenarios match actual
API contracts, checking ﬁxture consistency, and detecting missing integration test coverage.
Anti-pattern: Generating test code during the review phase. The TDD review agent should
produce a validation report on the test plan, not test implementations. Mixing review and
generation leads to the agent "ﬁxing" ambiguous criteria by making assumptions rather than
ﬂagging them.
The pre-commit agent must explain errors through structured output, never auto-ﬁx
Top recommendation: Implement the Pre-commit Check Agent using Claude Code's custom
mode or hook system with ﬁle-editing tools explicitly excluded. Output format should be
structured as ERROR → ROOT CAUSE → OPTIONS (with trade-offs), not conversational. Use
PreToolUse hooks to block Edit/Write tools entirely.
The explain-don't-ﬁx pattern is implemented in Cursor AI's "Ask Mode" (read-only, no code
changes),  JetBrains AI Assistant's "Explain with AI" feature,  and Claude
Code's custom modes with tool restrictions. The key architectural insight is mode separation:
explanation and modiﬁcation must be physically separated by restricting available tools, not by
instruction alone. System prompt instructions to "never modify ﬁles" are insuﬃcient — models
exhibit sycophantic drift and start suggesting ﬁxes despite instructions.

Hallucinated package detection is the highest-urgency security concern
Top recommendation: Integrate Socket.dev's MCP server (https://mcp.socket.dev/) for real-
time dependency scoring, combined with a veriﬁcation step that checks package existence,
download count, creation date, and maintainer history. Flag any package with fewer than 1,000
downloads, less than 6 months of history, or Levenshtein distance ≤ 2 from a known package
name.
The USENIX Security 2025 paper documents a 19.7% overall hallucination rate across 2.23
million package recommendations from 16 LLMs — 440,445 hallucinated packages including
205,474 unique non-existent names. Commercial models hallucinate at 5.2%, open-source at
21.7%.  Critically, 43% of hallucinated packages appeared consistently across repeated
prompts, making them predictable targets for slopsquatting attacks. 
The weaponization is already real. A hallucinated huggingface-cli package uploaded to PyPI
received 30,000+ downloads in 3 months after Alibaba copy-pasted the hallucinated install
command into a public repository.  The term "slopsquatting" (coined by Python PSF
Developer-in-Residence Seth Larson) now describes the attack pattern.  Andrew
Nesbitt's December 2025 research shows slopsquatting combining with dependency confusion
attacks, with 404 logs from package registries serving as "toxic waste" revealing which names
attackers should squat.

Performance diagnosis maps load test output to SLO burn rates, not raw thresholds
Top recommendation: Structure the Performance Diagnosis Agent around Google SRE's multi-
window burn rate alerting pattern rather than simple threshold checks. Input should include
p50/p95/p99 histograms, error rates, traﬃc volume, saturation metrics, and deployment markers
with commit hashes. Output should specify burn rate, time to error budget exhaustion,
suspected causes with evidence, and recommended remediation.
Datadog's Bits AI SRE is the most comprehensive production implementation — it forms
hypotheses, gathers telemetry iteratively, and correlates signals across logs, metrics, traces, and
source code.  Its "Suspect Commits" feature directly maps performance regressions to
speciﬁc code changes using deployment tracking. The pattern is: tag every deployment with
version/commit hash → compare golden signals between consecutive versions → identify the
commit window where regression appeared.
The standard input/output contract for p99-latency diagnostics follows this structure: SLO
deﬁnition (service, SLI metric, target, window, error budget)  → current metrics
(percentile histograms, error rate, traﬃc, saturation) → recent deploys with commit references →
triage logic (check burn rate, correlate with deployments, examine dependencies, check
saturation) → structured output (severity, burn rate multiplier, budget exhaustion ETA,
suspected causes ranked by conﬁdence, recommended actions).
k6 provides direct CI integration through its thresholds system — non-zero exit codes on
threshold breach that can block PR merges.  Grafana Cloud k6 now oﬀers AI-
powered insights connecting test outcomes to SLOs. Locust reports per-endpoint
p50/p90/p95/p99 with CSV export for downstream analysis.
Anti-pattern: Averaging percentiles across time windows. This is mathematically incorrect and
produces misleading results. Use raw histograms (OpenTelemetry bucket-based) for percentile
computation. Google SRE explicitly warns against percentile aggregation.
Specialist agents outperform generalists when scoped to fewer than 200 instructions
The empirical evidence on specialization vs. generalism converges on a clear ﬁnding: specialist
agents with focused, short system prompts outperform general-purpose agents with large
context dumps, but only when the task boundaries are well-deﬁned.
The AgentCoder ablation study (ACL) found that separating programmer and test designer roles
into distinct agents improved state-of-the-art performance. MapCoder (ACL 2024) conﬁrmed
that "every agent has its role in the pipeline as turning oﬀ any agent reduces performance."
 MALMM (IROS 2025) found that "a single agent performing multiple roles
struggles to excel in all of them" and proposed role-speciﬁc contexts.

The optimal system prompt length is constrained by two ﬁndings. First, HumanLayer's analysis

 Second, the "Same Task, More Tokens" paper (ACL 2024) found reasoning

Research Area 2: Reusable prompt skills
Socratic PRD elicitation requires enforced ask-before-generate boundaries
Top recommendation: Structure the PRD skill system prompt using the Flipped Interaction
Pattern: the agent asks questions one at a time before generating any content. Use breadth-ﬁrst
sequencing (sweep across user roles, then edge cases, then technical constraints) before depth-
ﬁrst drilling into any single area. Place the questioning behavior in the system prompt; provide
the output template in the user turn.
The most reliable enforcement mechanism is explicit behavioral gating: "You MUST ask at
least 8 clarifying questions across all user roles and edge cases before generating
any specification content. If you generate content before questions are resolved,
you have failed your task." Moxo's production agent prompts use: "Before executing any
action, verify all required inputs. If uncertain, ask for clarification rather than
guessing."  The Cognitive Veriﬁer Pattern (documented by Zapier) tells the AI to break
questions into smaller parts and get feedback before proceeding. 
Question sequencing should follow the Illumination → Confrontation → Clariﬁcation →
Integration cycle from the AI-Augmented Socratic Method.  Start with open
questions about user roles and intended workﬂows (breadth), then challenge assumptions with
"what happens if..." scenarios (depth), then clarify ambiguities, then synthesize. Critically, ask 2–
3 questions at a time — asking all questions at once overwhelms the developer, while one-at-a-
time is too slow.
ChatPRD is the most mature production tool implementing this pattern,  with MCP
integration for Claude Code and direct conversion to Linear/Jira tickets. Fireside PM's 2025
benchmark ranked Claude #1 among 5 AI tools for PRD quality, citing "strongest strategic
thinking."

Ticket generation should use tool calling, not JSON mode, with two-step reasoning
Top recommendation: Use API-level structured outputs via tool/function calling (not raw JSON
mode) for ticket generation. Implement a two-step approach: let the model reason freely about
the PRD ﬁrst, then convert reasoning into structured ticket format. Validate output against
Pydantic schemas before import.
Critical research ﬁnding from Tam et al. (EMNLP 2024 Industry Track): JSON-mode
constrained decoding degrades reasoning by 10–15% because models are forced to output
answers before completing chain-of-thought. The ﬁx is to always place reasoning ﬁelds before
answer ﬁelds in schemas, or better, separate the reasoning and structuring steps entirely.
The Instructor library benchmarks (2024) found that Claude models see signiﬁcantly greater
improvement with few-shot examples compared to GPT-4o variants for structured output.
 For batch ticket generation speciﬁcally, few-shot examples of 2–3 complete tickets in
the target format consistently outperform detailed format instructions alone.
Each ticket should contain: epic reference, title, type (story/task/bug), description, acceptance
criteria in Gherkin format, API endpoint speciﬁcations (path, method, query/path/body
parameter labels), response schema reference, estimated complexity, and dependency
references. The PRD-to-ticket pipeline documented by Product Direction adds a reﬁnement
agenda — categorizing items as ALREADY DECIDED (from PRD), VALIDATE WITH TEAM
(technical decisions), or OPEN QUESTIONS (need input) — which prevents the agent from
making assumptions about unresolved issues. 
No native bulk creation exists in Linear MCP. The create_issue tool processes one issue at a
time. For a PRD generating 15–30 tickets, the agent must call create_issue iteratively. This is a
meaningful constraint — plan for sequential creation with parent-child relationships established
via parentId.
Anti-pattern: Forcing JSON structure during the reasoning phase. Let the model analyze the
PRD and identify ticket boundaries in natural language ﬁrst, then format into the target schema.
Also avoid generating tickets from partial PRDs — incomplete input produces speculative
acceptance criteria.

Anthropic's engineering team provides the authoritative guidance: "The core challenge of long-
running agents is that they must work in discrete sessions, and each new session begins with
no memory of what came before."  Their recommended architecture uses an initializer
agent plus a coding agent that "makes incremental progress in every session, while leaving clear
artifacts for the next session." 
The Document and Clear pattern (widely adopted in the Claude Code community) is: have
Claude dump its plan and progress into a .md ﬁle, /clear the context, then start a new session
pointing at that ﬁle. Trail of Bits' claude-code-conﬁg repository formalizes this: "commit your
work, write a brief plan to a ﬁle, /clear, and start the next session by pointing Claude at that ﬁle.
A fresh session with a written handoﬀ is usually better than resuming a stale one — the context
is cleaner and the prompt cache is warm." 
Blake Link's Session Handoﬀ Protocol provides the most detailed documented implementation:
context-aware titles with measurable goals, quantiﬁable progress metrics, prioritized focus areas,
recent accomplishments (preventing rework), development environment details, and speciﬁc
expected outcomes.  His key insight: "Generic handoﬀs are worse than no handoﬀs.
'Continue working on the API' is less helpful than '30/90 tests passing, focus on webhook
retry mechanisms in test_advanced_features_and_monitoring.py.'"
Anti-pattern: Relying on auto-compaction (/compact) for continuity. Multiple practitioners
warn against this — Shrivu Shankar (Abnormal AI) states: "I avoid this as much as possible. The
automatic compaction is opaque, error-prone, and not well-optimized."  Two
compactions in a session signals the task was too large.  SitePoint's guide
recommends compacting at 60% context, not 90%, and limiting sessions to work expected to
consume under 120K input tokens. 
AI-generated code review must address the 8x duplication and 2.7x security gap
Top recommendation: Structure the C.L.E.A.R. review skill around ﬁve dimensions
(Correctness, Logic/License, Eﬃciency, Architecture, Reliability) with three AI-speciﬁc
additions: hallucinated dependency veriﬁcation, duplication detection against existing
codebase, and assertion depth validation in generated tests. Run automated pre-checks
(SonarQube, Semgrep, dependency veriﬁcation) before human review to reduce the review
surface.
The evidence for AI-speciﬁc review concerns is now quantitative. GitClear's 2025 study (211
million lines analyzed) found: refactoring collapsed from 25% of changed lines in 2021 to
under 10% in 2024, copy/pasted code rose from 8.3% to 12.3%, and duplicate code blocks rose
eightfold.  Short-term churn (code revised within 2 weeks) reached 7.9% in 2024 versus
5.5% in 2020.  CodeRabbit's analysis of 470 open-source PRs found AI-generated PRs
produced 10.83 issues per PR versus 6.45 for human-only, with 2.74× more security
vulnerabilities and signiﬁcantly more readability violations. 
Salesforce's Prizm system (January 2026) represents the most sophisticated production
response. As AI-generated PRs grew to regularly exceed 20 ﬁles and 1,000 lines, traditional ﬁle-
by-ﬁle review "fractured." Prizm reconstructs developer intent rather than reviewing ﬂat diﬀs,

AGENTS.md should be the universal source under 200 lines with progressive disclosure
Top recommendation: Use AGENTS.md as the canonical instruction ﬁle (60,000+ projects, 25+
tool integrations via the Linux Foundation's Agentic AI Foundation).  Keep the root
ﬁle under 200 lines. Use directory-speciﬁc AGENTS.md ﬁles for package/module-speciﬁc
instructions. Point to detailed docs in a docs/context/ or agent_docs/ subdirectory rather
than inlining content.  Symlink tool-speciﬁc ﬁles (CLAUDE.md, .cursorrules) to
AGENTS.md for cross-tool compatibility.
The progressive disclosure pattern is consistently recommended across Anthropic,
HumanLayer, Builder.io, and Trail of Bits. The root ﬁle covers WHAT (tech stack, project
structure), WHY (project purpose), and HOW (build/test/verify commands) — nothing else.
 Domain-speciﬁc instructions live in nested ﬁles that the agent discovers on demand.
 Claude Code's skills architecture (.claude/skills/ with SKILL.md ﬁles) implements
this at the tool level  — skill descriptions load at session start (~100 tokens each), full
content loads only when invoked. 
HumanLayer's critical ﬁnding: Claude Code wraps CLAUDE.md content with "IMPORTANT:
this context may or may not be relevant to your tasks."  This means Claude
actively deprioritizes instruction ﬁle content it deems irrelevant, making overstuﬀed ﬁles
counterproductive rather than merely wasteful. 
The amend-commit pattern for instruction ﬁle updates: modify only the speciﬁc section that
changed, commit the instruction ﬁle change alongside the code change that motivated it, use
descriptive commit messages. Trail of Bits' reference implementation demonstrates the
approach — global CLAUDE.md covers development philosophy and hard limits, while skills and
agents encode specialized expertise.

Research Area 3: MCP tool integration
Linear MCP exposes 25 tools but lacks bulk creation and blocking relationships
Top recommendation: Use the oﬃcial Linear MCP server at https://mcp.linear.app/mcp for
issue lifecycle management. Plan for iterative single-issue creation (no bulk API exists) and
manage blocking relationships through the underlying GraphQL API or a community server like
cosmix/linear-mcp which exposes relationship support.
The oﬃcial server exposes 25 tools covering issues, projects, comments, documents, cycles,
teams, and users.  It uses OAuth 2.1 with dynamic client registration for interactive
auth and supports API key Bearer tokens for programmatic access.  Rate limits are 1,500
requests/hour per user with API key auth (250,000 complexity points/hour) using a leaky
bucket algorithm. 
Key capability gaps for the described workﬂow:
Blocking relationships (issueRelationCreate for blocked-by/blocking) are available in
the Linear GraphQL API but not directly exposed as MCP tool parameters
Custom ﬁelds are not explicitly surfaced in the MCP create_issue tool
Bulk issue creation does not exist — even the underlying GraphQL API processes issues
one at a time via issueCreate
For the PRD-to-Issues skill generating 15–30 tickets, the agent will need to call create_issue
sequentially, establishing parent-child relationships via parentId and handling labels via
labelIds to encode metadata (path, HTTP method, parameter types).  Structured
metadata beyond native ﬁelds must be embedded in the issue description using consistent
formatting.
The February 2026 update added support for initiatives, project milestones, and project updates
— useful for higher-level project management integration but not directly addressing the ticket-
level workﬂow gaps. 
Anti-pattern: Assuming the MCP surface equals the full API. The Linear MCP server exposes a
curated subset of the GraphQL API.  Teams needing blocking relationships, custom ﬁelds,
or webhook conﬁguration must use the GraphQL API directly or a community server. The
cosmix/linear-mcp community server adds relationship support and health status tracking.

Top recommendation: Use PG-MCP (stuzero/pg-mcp-server) for development and staging
environments — it is the only MCP server with explicit pgvector awareness through YAML-

MCP invocation should use dynamic loading with code-execution for data-heavy operations
Top recommendation: Default to reactive/dynamic tool loading rather than front-loading all
MCP tools at session start. Use Claude Code's built-in Tool Search feature (limits tool deﬁnitions
to 10% of context), enable toolset ﬁltering on all MCP servers, and use the code-execution
pattern for data-heavy operations to achieve 98%+ token savings on results.
The core problem is quantiﬁed: each MCP tool deﬁnition costs 550–1,400 tokens on every API
call regardless of use. One practitioner reported 3 MCP servers consuming 143,000 of 200,000
tokens (72%) before any conversation began. A Scalekit benchmark showed MCP costing 4–32×
more tokens than CLI for identical operations.
Anthropic's recommended approach is the code-execution pattern (documented November
2025): instead of direct tool calls, the agent writes code that calls MCP tools, ﬁlters results in the
execution environment, and returns only relevant data to the model context. Example: 150,000
tokens of raw database query results become 2,000 tokens after code-based ﬁltering — a 98.7%
reduction. This pattern also provides privacy beneﬁts, as intermediate results never enter the
model's context.
For this workﬂow speciﬁcally, the session-type-based loading pattern is recommended:
Planning sessions: Load Linear MCP + Figma MCP, disable GitHub and PostgreSQL
Implementation sessions: Load GitHub MCP (repos + actions toolsets) + PostgreSQL MCP,
disable Linear and Figma
Review sessions: Load GitHub MCP (pull_requests + code_security) + Linear MCP, disable
Figma and PostgreSQL
Claude Code's feature loading hierarchy provides the mechanism: CLAUDE.md and MCP
servers load at session start (expensive), skills load only descriptions initially (cheap, ~100
tokens each), subagents run in isolated context (no main-context cost), and hooks execute
externally at zero context cost. The Context Mode MCP Server (mksglu/context-mode)
virtualizes tool output to reduce consumption by 98%, extending session time from ~30 minutes
to ~3 hours.
Anti-pattern: Front-loading all 4 MCP servers in every session. With Linear (25 tools), Figma (12
tools), GitHub (70+ tools), and PostgreSQL (varies by server) loaded simultaneously, tool
deﬁnitions alone could consume 50,000+ tokens before any work begins, degrading model
performance on the actual task.

chain risk with immediate ROI.
5. Pre-commit Check Agent — conﬁgure Claude Code custom mode with Edit/Write tools

Open questions where industry consensus does not exist
No controlled specialist-vs-generalist comparison exists for production software
engineering. The AgentCoder and MapCoder ablation studies demonstrate specialist beneﬁts
within their speciﬁc architectures, but no published study directly compares the same task
solved by a single agent with comprehensive context versus multiple specialist agents with
focused context in a controlled production setting. Anthropic's guidance to "start simple" and
the Agentless approach's competitive performance suggest the answer may be task-dependent.
The optimal system prompt length is empirically bounded but not precisely deﬁned. The
150–200 instruction limit from HumanLayer's analysis cites a single preprint (arXiv 2507.11538)
not yet widely replicated. The 3,000-token reasoning degradation threshold (Levy et al., ACL
2024) applies to reasoning tasks with padding, not speciﬁcally to system prompts. Practitioners
converge on "under 300 lines" but the exact sweet spot varies by task complexity.
CLAUDE.md system-reminder wrapping behavior is documented by proxy analysis but not
oﬃcially conﬁrmed by Anthropic. HumanLayer's ﬁnding that Claude Code injects a "may or
may not be relevant" wrapper around CLAUDE.md content is consequential for instruction ﬁle
design but has not appeared in Anthropic's public documentation.
TDD review at the planning gate lacks production tooling. No production-grade, widely-
adopted tool speciﬁcally validates test plans before implementation. The Meridian RFC and
requirements veriﬁcation megaprompts represent early approaches, but the gap between
academic acceptance criteria validation and production TDD review remains open.
MCP prefetch patterns have no formal speciﬁcation. While Anthropic documents the code-
execution pattern and Claude Code's Tool Search feature, there is no published standard for
session-type-based MCP loading proﬁles or a formal "MCP prefetch" speciﬁcation analogous to
continuation prompts for ﬁle context. Current approaches (mcpick, --toolsets, environment-
scoped conﬁgs) are ad-hoc.
Database MCP security for multi-environment workﬂows lacks a mature access control
model. While all practitioners agree on never connecting to production, the mechanisms for
enforcing this (environment-scoped tool whitelisting, connection ID isolation, network
segmentation) are implemented diﬀerently across servers with no standard approach. The
Cerbos and MintMCP approaches to tool-level permission scoping are promising but not yet
widely adopted.
