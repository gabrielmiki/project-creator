Specialist review agents for AI-assisted development workﬂows
Six specialist agents—architecture, security review, TDD validation, pre-commit
explanation, security diagnosis, and performance diagnosis—should each operate as isolated
Claude Code subagents with XML-structured prompts under 2,500 tokens, terminal-tool
output contracts enforcing structured JSON, and phase-gated invocation tied to your 8-phase
methodology. The evidence base from Stripe (1,300+ AI-authored PRs/week), 
Ellipsis (27 months in production),  Shopify (ICML 2025),  and Anthropic's
own subagent architecture converges on a single design principle: many small specialists with
restricted tool access outperform one mega-agent every time.  This report delivers prompt
structures, invocation triggers, input/output contracts, failure modes, and MCP tooling
recommendations for each agent, grounded in 2024–2026 production evidence.
The tech stack—FastAPI, SQLAlchemy 2.0, PostgreSQL with pgvector, Redis, Celery, LangGraph,
Linear, Figma, GitHub—has mature MCP server coverage across every integration. The critical
gap is hallucinated dependency detection, where 19.7% of LLM-recommended packages are
fabricated  (USENIX Security 2025),  but MCP-compatible tools now exist to
close it.
The subagent architecture that actually works in production
Claude Code's subagent system (v2.1.68+) provides the correct primitive for specialist review
agents.  Each subagent receives only its custom system prompt plus basic environment
details—not the full Claude Code system prompt.  This isolation prevents context
pollution, the single largest failure mode in production agent systems.

Second, enforce structured output via a terminal tool pattern. Archbot's production system
requires the model to call a submit_review function with typed ﬁelds  (severity
enum, ﬁle/line citation, evidence string, suggestion) rather than producing freeform Markdown.
This makes it harder to hand-wave.  Ellipsis independently converged on the same pattern.
When the model must populate ﬁelds rather than write prose, false positive rates drop
measurably because the model is forced to cite evidence. 
Third, keep prompts between 1,500 and 2,500 tokens. GitHub Copilot hard-caps custom
instructions at 4,000 characters.  Cursor caps at 6,000 characters per rule ﬁle. 
Claude Code's built-in veriﬁcation specialist runs at ~2,453 tokens.  Anthropic's
"progressive disclosure" principle—show just enough, then reveal more on demand—is the right
mental model.  If your agent prompt exceeds 3,000 tokens, you are almost certainly
mixing concerns.
Anti-pattern: the growing system prompt. Shopify's ICML 2025 paper documents "Death by a
Thousand Instructions"—as their tool count grew past 20, system prompt bloat degraded
performance. Past 50 tools, the model couldn't reliably distinguish between tools doing similar
things. Their ﬁx: JIT (Just-in-Time) instructions delivered alongside tool responses instead of
front-loaded in the system prompt.  For review agents speciﬁcally, this
means embedding context in the input payload, not the system prompt.

Why static analysis output matters as input: The LLM adds value that static analysis alone
cannot: it understands business logic context. Bandit ﬂags all execute() calls; the LLM can
distinguish between a parameterized query in a repository method and a raw SQL string built
from user input. Endor Labs (2025) found that traditional SAST tools miss "changes that are
technically valid but shift your security architecture"—like a new unauthenticated endpoint on a
destructive operation. The LLM catches architectural security changes.
Critical ﬁnding: A March 2026 study of Claude Code, OpenAI Codex, and Google Gemini
building applications found all three repeat decade-old security mistakes. Broken access
control was universal. Unauthenticated endpoints on destructive operations appeared
consistently across all three agents. This is why the security review agent exists: the coding agent
itself will introduce these vulnerabilities.

Agent 3: TDD review needs both stories and conftest
Invocation trigger: Before execution of any implementation ticket. This agent validates the test
plan, not the implementation. Fire it after the story/spec is written but before the developer
(human or AI) begins coding.
The answer to "story ﬁles alone or conftest context" is unambiguously both, for diﬀerent
validation tasks:
Story ﬁles alone validate whether acceptance criteria are testable. The agent checks: Are criteria
binary (pass/fail)? Are they in Given/When/Then format? Do they cover happy path, error cases,
and edge cases? Does each criterion contain concrete values or thresholds that can be asserted
against? "The system should be fast" fails validation. "Response time < 200ms at p95" passes.
Conftest architecture context validates whether the test infrastructure can actually support the
proposed test plan. The agent checks: Do required factory_boy factories exist for all model
dependencies? Are SubFactory chains complete (e.g., BookFactory → AuthorFactory)? Are
GitHub
ﬁxtures registered at the correct scope? Are there circular dependencies in the factory graph?
Does the database session ﬁxture conﬁguration support the proposed test isolation strategy?

Test ordering validation in practice: The TDAD framework (arXiv, 2026) builds a code–test
dependency graph from the repository—mapping which tests should run for which code
changes.  For the TDD review agent, the simpler version is suﬃcient: validate that
proposed tests follow unit → integration → E2E ordering, that no test depends on side eﬀects
from a previous test, and that database ﬁxtures use function scope for isolation.

Agent 4: Pre-commit explanation beats silent auto-ﬁxing
Invocation trigger: Pre-commit hook or manual invocation after implementation, before
commit. Runs ruﬀ, mypy, pytest, pip-audit, and bandit as a quality gate.
The design decision is explain-don't-ﬁx, and the evidence is clear. Microsoft's AgentRx
framework  (2026) demonstrated +23.6% improvement in failure localization
accuracy  using structured explanation over conversational LLM prompting. Developer
trust in AI tools is declining (Stack Overﬂow 2025: 43% → 33% trust AI accuracy),  and
auto-ﬁxing without explanation accelerates this erosion. Developers who don't understand why
a ﬁx was applied will revert it or introduce the same pattern elsewhere.
The structured format wins over conversational. Each ﬁnding should follow: error → root cause
→ options with trade-oﬀs. Not "here's what's wrong and here's the ﬁx" but "here's what's wrong,
here's why it's wrong at a conceptual level, and here are 2-3 ways to address it with diﬀerent
eﬀort/safety trade-oﬀs."

Agent 5: Hallucinated package detection is a solved MCP problem
Invocation trigger: Post-implementation, after any ticket that introduces new dependencies.
Also runs as part of the pre-commit quality gate.
The threat is real and quantiﬁed. The USENIX Security 2025 study tested 16 LLMs generating
576,000 code samples and found 19.7% of all recommended packages were hallucinated—
440,445 total instances, 205,474 unique fabricated names. 43% of hallucinated packages
repeated consistently across 10 re-queries, making them exploitable.  Attackers register
these names preemptively—a practice now called "slopsquatting"  (coined by
Seth Larson, Python Security Developer-in-Residence).
The combined attack surface is worse than either dependency confusion or hallucination alone.
 Andrew Nesbitt's analysis (December 2025) explains: LLMs hallucinate
consistently based on training data patterns that include leaked internal code. An attacker
squats hallucinated names, and some will match real internal package names. This inverts the
discovery problem that traditionally made dependency confusion attacks diﬃcult.
MCP-compatible detection tools exist today:
The sinewaveai/agent-security-scanner-mcp is the most comprehensive option. Its
scan_packages tool checks imports against 4.3M+ known packages across 7 ecosystems (PyPI,
npm, RubyGems, crates.io, pub.dev, CPAN, raku.land).  It detects hallucinated
packages like flask-async-helpers while conﬁrming real packages like flask, sqlalchemy,
pydantic.  Install with npx agent-security-scanner-mcp init claude-code.
The mcp-pypi server (v3.0.0, pip install mcp-pypi) is security-focused: it searches
packages, scans for vulnerabilities, audits dependency trees, and checks vulnerabilities before
suggesting packages. It can detect transitive dependency vulnerabilities. 
The pypi-query-mcp-server provides get_package_versions—critical for verifying that a
package exists at a speciﬁc pinned version.  A package might exist but not at the version
the LLM hallucinated.

Anti-pattern: relying solely on PyPI lookup for safety. RAG-based veriﬁcation against live
registries oﬀers a "false sense of security" because an adversary can trivially register a new
package.  The correct layered approach is: (1) check package existence, (2)
check package age (ﬂag anything registered < 90 days before the model's training cutoﬀ), (3) use

Conclusion: the patterns that separate production agent systems from demos
The most important insight across all six agents is that the terminal tool pattern—forcing
structured output via a submit action rather than freeform text—is the single highest-
leverage design decision. Every production system surveyed (Archbot, Ellipsis, Claude Code's
built-in veriﬁcation specialist, Keptn quality gates) converges on this. Freeform prose lets models
sound conﬁdent about wrong ﬁndings. Structured ﬁelds with citation requirements force
accountability. 
The second insight is that agent scope determines agent quality more than prompt
engineering does. Shopify learned this with 50+ tools degrading performance.  Ellipsis
learned it by splitting one mega-agent into dozens of specialists.  The architecture
reviewer should never see security concerns. The security reviewer should never comment on
test ordering. Scope restriction via Claude Code's tools ﬁeld (limiting to read-only for review
agents) is a mechanical enforcement of this principle. 
The third insight concerns trust calibration. Developer trust in AI tools is declining—43% to 33%
between 2024 and 2025—and false positives are the primary cause.  CodeAnt AI's multi-
model consensus approach (ﬂag only when 2+ of 3 models agree) cuts false positives ~60% while
maintaining 92% recall.  Ellipsis's feedback-based ﬁltering (embedding search over past
developer reactions) provides continuous improvement.  The explain-don't-ﬁx pattern
preserves developer agency. These are not optional polish—they determine whether the agent
system survives contact with a real engineering team.
The hallucinated dependency problem is the most underappreciated risk in LLM-assisted
development. Nearly one in ﬁve package suggestions is fabricated, and the slopsquatting attack
surface means those fabrications are actively weaponized.  The agent-security-
scanner-mcp tool should be considered mandatory infrastructure, not optional tooling. 
 Combined with hash-pinned lockﬁles and isolated installation environments, 
this closes the most dangerous gap in the workﬂow.
