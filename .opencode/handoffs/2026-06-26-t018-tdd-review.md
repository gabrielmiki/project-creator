## 1. Executive Summary

T-018 targets the most comprehensive integration tests in the project: multi-plugin combinations, GUI worker lifecycle, overwrite flow, and error/rollback scenarios. While the ticket follows established T-016/T-017 patterns and has well-structured test areas, it contains **4 blocking issues** and **7 moderate issues** that must be resolved before implementation. The most critical problems are: (1) the `all_plugins_spec` fixture is impossible under the current domain model, (2) the `worker.error` signal test relies on a dead signal wire, (3) file conflict tests cannot work with built-in plugins alone, and (4) GUI worker fixtures need `qapp` access from integration conftest.

- **Status**: âš ď¸Ź NEEDS FIX

## 3. Structured Findings

### 3.1 Blocking Issues

#### B1. `all_plugins_spec` fixture is impossible under current domain model

| Attribute | Detail |
|-----------|--------|
| **Source** | Fixture definition: `all_plugins_spec(spec_factory) -> ProjectSpec with all available plugins somehow` |
| **Problem** | `ProjectSpec.template` has only `backend_id: str` and `frontend_id: str | None`. There is no mechanism to represent "all 4 plugins" (fastapi + django + react + htmx) in a single spec. The domain model provides exactly 2 plugin slots. The "somehow" in the ticket description confirms this is unresolved. |
| **Impact** | This fixture cannot be implemented without either (a) changing the domain model, (b) running generation multiple times per test, or (c) redefining the scope to test pairwise combos. |
| **Suggested fix** | Replace the "all plugins" concept with explicit pairwise combos. Define `full_spec` as `backend_id="fastapi"` + `frontend_id="react"` with config for both. Define `django_htmx_spec` as `backend_id="django"` + `frontend_id="htmx"`. The `all_plugins_spec` fixture should be removed or redefined as a spec factory that accepts a list of plugin IDs (though this still can't be represented in ProjectSpec without domain changes). |

#### B2. `test_worker_error_signal_on_failure` tests a dead signal wire

| Attribute | Detail |
|-----------|--------|
| **Source** | Test table: `test_worker_error_signal_on_failure` â€” "worker.error emitted when generation fails" |
| **Problem** | `GenerationWorker.error = Signal(str)` is defined at `workers.py:55` but **NEVER emitted in `run()`** (lines 76-94). The `run()` method catches exceptions, calls `txn.rollback()`, and emits `finished.emit(result)` â€” it does NOT emit `error`. The signal wire is dead. Both T-013 post-mortem (Â§7) and T-015 post-mortem (Â§7) list this as an outstanding issue: "GenerationWorker.error = Signal(str) is NEVER emitted â€” dead signal wire." |
| **Impact** | A test asserting `worker.error.emit(...)` is called will fail. If the test asserts `worker.error` signal is connected, that works â€” but the signal is never emitted, so `QSignalSpy(worker.error).count()` would remain 0 after generation failure. |
| **Suggested fix** | Either (a) implement the dead signal wire: modify `GenerationWorker.run()` to emit `self.error.emit(str(e))` before `finished.emit(result)` in the except block, or (b) remove the `error` signal test and test the actual behavior (finished emitted with success=False). Option (a) is preferred since the signal is already defined and connected in MainWindow. |

#### B3. File conflict tests assume conflict sources that don't exist in built-in plugins

| Attribute | Detail |
|-----------|--------|
| **Source** | Test table: `test_plugin_file_conflict_warning` â€” "Two plugins writing to same relative path: last-writer-wins, warning logged" |
| **Problem** | The 4 bundled plugins produce naturally non-conflicting file trees: FastAPI â†’ `app/`, Django â†’ `config/` + `apps/`, React â†’ `src/`, HTMX â†’ `templates/`. The only partial overlap is between Django and HTMX (both produce `templates/` and `static/` directories, but not the same *files*). `PluginExecutionEngine` stages files through `txn.stage_file()` which overwrites by Path. `txn.stage_directory()` creates dirs idempotently (exist_ok=True). No warnings are logged for directory overlaps â€” only files that share the same exact relative path would trigger a warning, and no two bundled plugins produce the same file. |
| **Impact** | The test would require either (a) injecting a custom plugin that intentionally conflicts with a built-in plugin, or (b) verifying that no warning is logged (which adds no value). The test as described tests a scenario that doesn't occur with the bundled plugins. |
| **Suggested fix** | Clearly define whether this test uses (a) a custom conflicting plugin injected into the registry, or (b) Django + HTMX directory overlap (which is expected and not an error). If (a), specify the test-double plugin. If (b), rephrase as: "Django and HTMX plugins both create templates/ directory â€” verify no errors and both sets of files appear." |

#### B4. Worker fixture missing `qapp` dependency and `txn` strategy

| Attribute | Detail |
|-----------|--------|
| **Source** | Fixture: `worker(orchestrator, full_spec, temp_dir) -> GenerationWorker` |
| **Problem** | `GenerationWorker` extends `QObject`, which requires a `QApplication` instance. The fixture signature includes `orchestrator`, `full_spec`, and `temp_dir` but omits `qapp`. Without `qapp`, tests will crash with "QApplication must be created before QWidgets" (same issue as T-012 Round 1 B-001). Additionally, the fixture doesn't specify whether `txn` should be a real `GenerationTransaction` or a `MagicMock` â€” this matters because real transactions use real filesystem I/O. |
| **Impact** | The fixture will fail at collection time (if PySide6 is imported without qapp) or at runtime. |
| **Suggested fix** | (a) Add `qapp` as a fixture requirement. (b) Document the txn strategy: integration tests should use real `GenerationTransaction` (matching T-016 pattern), not `MockTransaction`. (c) The worker fixture must also handle `progress` and `overwrite_confirmed` parameters. |

### 3.2 Moderate Issues

#### M1. `test_worker_signals_connected` needs Passive GenerationScreen model

| Source | Test table: `test_worker_signals_connected` |
|--------|---------------------------------------------|
| **Problem** | T-015 established that `GenerationScreen` is **PASSIVE** â€” it has no direct signal connections from the worker. `MainWindow` owns ALL signal routing: `worker.finished â†’ _on_generation_finished â†’ gen_screen.on_finished()`. The test must verify signal connections through `MainWindow`, not through `GenerationScreen` directly. |
| **Impact** | If the test connects worker signals directly to GenerationScreen (bypassing MainWindow), it tests a non-existent architectural pattern. |
| **Suggested fix** | Rephrase to: "Given a MainWindow with a GenerationWorker, when signals are wired by `_create_generation_worker()`, then worker.finished/progress/log/error are connected to MainWindow handlers." Verify via `QSignalSpy` on `generation_completed` (MainWindow signal), not directly on worker signals. |

#### M2. `test_error_missing_plugin_id` confuses validation error with exit code

| Source | Test table: `test_error_missing_plugin_id` â€” "CLI spec with non-existent plugin_id: error returned, exit code 1" |
|--------|-------------------------------------------------------|
| **Problem** | The test name implies CLI testing ("exit code 1") but the description says "error returned." These are different paths: (a) CLI headless path: `ValidationEngine.validate_spec()` returns `ValidationError`, then `_run_headless()` calls `sys.exit(1)` â€” exit code 1. (b) Orchestrator.generate() path: `ValidationEngine` is NOT called â€” the unresolved `backend_id` causes `PluginRegistry.resolve()` to raise `KeyError`, caught by orchestrator's try/except, returns `GenerationResult(success=False)`. The test must specify which path is tested and assert accordingly. |
| **Impact** | An ambiguous test spec could produce wrong assertions. |
| **Suggested fix** | Clarify whether this is a CLI test (patching sys.argv, asserting SystemExit) or an orchestrator/generate test (asserting GenerationResult.success is False). Suggestions: split into two tests, or pick one path and explicitly document the choice. |

#### M3. `test_error_special_chars_in_name` â€” sanitization behavior may not exist

| Source | Test table: `test_error_special_chars_in_name` â€” "Special chars sanitized: warning logged, project created with sanitized name" |
|--------|-------------------------------------------------------|
| **Problem** | Neither `ValidationEngine` nor any other component in the pipeline has special-char sanitization logic. `ValidationEngine.validate_spec()` checks `project_name` is non-empty (line 26-33 of validation.py), but there's no sanitization or warning for special characters. The ticket implies behavior that doesn't exist. |
| **Impact** | The test would fail (no warning logged, no sanitized name created). |
| **Suggested fix** | Three options: (a) Implement special-char sanitization before this ticket (significant scope increase), (b) Change the test to verify `ValidationError` is returned for names with special chars (if that's the desired behavior), or (c) Defer to a future ticket. Option (b) is recommended: "Given a project_name with special characters, when validate_spec() is called, then a ValidationError with severity='error' is returned." |

#### M4. `test_plugin_dependency_ordering` â€” no real plugins have `requires` set

| Source | Test table: `test_plugin_dependency_ordering` â€” "If plugin A requires B, B runs first" |
|--------|-------------------------------------------------------|
| **Problem** | None of the 4 bundled plugins have `requires` or `run_after` set (all are `[]`). The topological sort in `PluginRegistry.topological_sort()` will return plugins in their registration order. A test verifying dependency ordering would need to inject a custom plugin with `requires` set. |
| **Impact** | If implemented without custom plugins, the test would trivially pass (any order satisfies an empty dependency graph), providing no verification of the dependency resolution logic. |
| **Suggested fix** | Either (a) inject a test-double plugin with `requires=["fastapi"]` and verify ordering, or (b) change to test `PluginRegistry.topological_sort()` directly with known dependency graph (this is already tested in T-016 integration tests as `test_plugin_discovery.py` covers topo sort). Option (b) suggests this test duplicates existing coverage. |

#### M5. `qapp` fixture not available in integration test conftest

| Source | Proposed `tests/integration/conftest.py` extension |
|--------|----------------------------------------------------|
| **Problem** | The `qapp` fixture is defined in `tests/unit/conftest.py` (session-scoped). Integration tests that create `QObject` subclasses (GenerationWorker, QtProgressReporter, QSignalSpy) need a `QApplication` instance. Currently, `tests/integration/conftest.py` has no `qapp` fixture. |
| **Impact** | Any integration test that imports from `forge.ui.workers` or `PySide6.QtTest` will crash without a qapp fixture. |
| **Suggested fix** | (a) Add `qapp` fixture to `tests/integration/conftest.py` using the same `QApplication.instance() or QApplication(sys.argv)` pattern. (b) Document `QT_QPA_PLATFORM=offscreen` for CI. (c) Consider extracting the qapp fixture to a shared conftest to avoid duplication with `tests/unit/conftest.py`. |

#### M6. CLI error tests need `pytest.raises(SystemExit)` and `capsys` patterns

| Source | Test table: `test_error_missing_plugin_id`, `test_error_missing_plugin_id` (CLI path) |
|--------|-------------------------------------------------------|
| **Problem** | CLI error tests that verify exit code 1 need `pytest.raises(SystemExit)` pattern. Tests that verify error messages need `capsys`. Neither pattern is documented in the ticket. T-017's TDD review (M1, N3, N4) found the same gap and documented `pytest.raises` pattern. |
| **Impact** | Tests may be implemented without correct SystemExit handling, producing silent passes or false failures. |
| **Suggested fix** | Document that CLI tests use `with pytest.raises(SystemExit) as exc_info:` for exit code assertions and `capsys.readouterr()` for message assertions. |

#### M7. Scaffold + files() overlap not tested

| Source | Coverage gap |
|--------|-------------|
| **Problem** | T-010 post-mortem (Â§7, Â§11.3) explicitly lists: "LOW: Scaffold + files() overlap is novel â€” no test verifies the overwrite semantics actually work (that would require an integration test beyond unit scope)." T-018 is the natural ticket for this integration test, but the test areas don't address it. When `react files()` AND `create-vite scaffold` both produce `src/App.tsx`, the staging overwrite must handle the duplication safely. No test verifies this. |
| **Impact** | If staging overwrite fails for simultaneous-writer patterns, the integration gap goes undetected until runtime. |
| **Suggested fix** | Add a test area: `test_scaffold_files_overlap_resolution` â€” Given a React plugin with bundler=vite, when generator runs, then src/App.tsx exists (from either files() or scaffold) with correct content. |
