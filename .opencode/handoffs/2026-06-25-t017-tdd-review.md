## 3. Structured Findings

### 3.1 Blocking Issues

#### B1. `test_orchestrator_empty_registry_warning` tests non-existent behavior

| Attribute | Detail |
|-----------|--------|
| **Source** | Test table: `test_orchestrator_empty_registry_warning` |
| **Problem** | "Empty registry in headless path shows warning (deferred from T-007)" — but the T-007 post-mortem (§7) explicitly lists this as **not implemented**: "No empty-registry warning in headless path — ticket testing notes mention it but implementation silently continues." The current `_run_headless()` code uses `except KeyError: pass` which silently swallows unresolved plugins. `ValidationEngine.validate_spec()` produces an `error` (not a warning) for unresolvable backend_id. There is no warning mechanism to test. |
| **Impact** | This test **cannot pass** without implementing the feature first. Either the feature must be scoped into T-017 (making it an implementation + test ticket) or deferred further. |
| **Suggested fix** | Options: (a) If this is meant to test `ValidationEngine.validate_spec()` returning an error for unresolvable IDs — rephrase as error test, not warning test. (b) If a true warning is desired (e.g., when `registry.discover()` returns empty), implement the warning in `_run_headless()` first, then test it. (c) Defer to a future ticket with an explicit note. |

#### B2. `test_orchestrator_rollback_on_failure` asserts wrong rollback behavior

| Attribute | Detail |
|-----------|--------|
| **Source** | Test table: `test_orchestrator_rollback_on_failure` |
| **Problem** | Describes: "only `.forge-staging` remains for diagnostics" — but `GenerationTransaction.rollback()` (line 79-87 of `transaction.py`) calls `shutil.rmtree(self.staging)` which **deletes** `.forge-staging`. The staging directory does NOT remain after rollback. |
| **Impact** | A test asserting `.forge-staging` exists after rollback would fail. A test asserting it does NOT exist would pass but contradict the ticket description. |
| **Suggested fix** | Correct the description to: "Stage failure triggers rollback, output dir is clean, `.forge-staging` is removed (no stale files)." The actual assertion should be `assert not (output_dir / ".forge-staging").exists()`. |

### 3.2 Moderate Issues

#### M1. `PluginExecutionEngine` integration tests may trigger real subprocess commands

| Attribute | Detail |
|-----------|--------|
| **Source** | `test_fastapi_plugin_execution_engine_real_plugin`, `test_orchestrator_full_pipeline` |
| **Problem** | The `FastapiPlugin` implements `CommandRunner`. Its `generate()` method calls `executor.run(["uv", "add", ...])`. The `PluginExecutionEngine.__init__` defaults to `ProcessExecutor()`. If tests don't inject a mock executor, they will execute real `uv add` commands — requiring `uv` to be installed, network access, and adding ~3s+ per test. T-016's pattern says "integration tests use REAL implementations" but also "mocking is limited to..." — running real subprocess commands in every pipeline test is slow and fragile. |
| **Impact** | Tests become slow (~3-5s per test), flaky (network-dependent), and fail without `uv`. |
| **Suggested fix** | Add a `mock_executor` fixture or accept `executor` injection in the `orchestrator` fixture. Document that the orchestrator fixture injects a no-op executor to avoid real subprocess calls, while all other components (registry, transaction, plugin) remain real. Example: `Orchestrator(real_registry, validation, executor=MagicMock(spec=ProcessExecutor))`. |

#### M2. `StdoutProgressReporter` vs `MockProgressReporter` inconsistency

| Attribute | Detail |
|-----------|--------|
| **Source** | AC-2 text vs fixtures section |
| **Problem** | AC-2 specifies `StdoutProgressReporter` but the fixture is `MockProgressReporter`. Both implement the `ProgressReporter` protocol, so tests work either way. But this creates confusion about which one to actually use. |
| **Impact** | Low — both work. But the AC should match fixture design. |
| **Suggested fix** | Align on `MockProgressReporter` (matching the fixture). Change AC-2 to reference `MockProgressReporter`. Rationale: integration tests don't need stdout output — assertions use the reporter's `.calls` list. |

#### M3. `real_registry` fixture duplication risk

| Attribute | Detail |
|-----------|--------|
| **Source** | Fixtures section (adding to conftest) vs existing `test_validation.py` |
| **Problem** | The ticket proposes adding a module-scoped `real_registry` to `tests/integration/conftest.py`. But `test_validation.py` already has its own module-scoped `real_registry` fixture (line 29-33). Adding the same name to conftest would create two fixtures with the same name. Pytest resolves this via shadowing (the module-level one wins for `test_validation.py`), causing confusion about which is canonical. |
| **Impact** | Low — tests still work. But creates maintainability confusion. |
| **Suggested fix** | Either (a) remove the local `real_registry` from `test_validation.py` and promote to conftest, or (b) name the conftest version something different (e.g., `discovered_registry`). Note: if removed from `test_validation.py`, check that no other test file depends on the module-scoped behavior. |

#### M4. `test_orchestrator_empty_project` description inaccuracy

| Attribute | Detail |
|-----------|--------|
| **Source** | Test table: `test_orchestrator_empty_project` |
| **Problem** | Describes: "Zero domains skips PluginExecutionEngine, produces shared structure" — but `PluginExecutionEngine.run()` skips based on `plugin_ids` (derived from `backend_id`/`frontend_id`), NOT based on `domains`. A spec with domains but no backend_id still skips PluginExecutionEngine. A spec with a backend_id but zero domains still runs PluginExecutionEngine. The two concepts are independent. |
| **Impact** | Low — the test would work correctly (using `backend_id=""`), but the description could mislead developers about the branching logic. |
| **Suggested fix** | Rephrase to: "`backend_id=""` (no plugins selected)/ skips PluginExecutionEngine, produces shared structure only." |

#### M5. `test_fastapi_plugin_execution_engine_real_plugin` underspecified assertions

| Attribute | Detail |
|-----------|--------|
| **Source** | Test table: `test_fastapi_plugin_execution_engine_real_plugin` |
| **Problem** | "Produces correct staged files and deps" is vague. "Deps" is ambiguous — does it mean `txn.requirements` (populated by `DependencyProvider.dependencies()`) or `requirements.txt` (a `GeneratedFile` from `FileProvider.files()`)? These are different mechanisms: dependencies go to `txn.requirements.extend(...)`, while `requirements.txt` is a file staged via `txn.stage_file()`. |
| **Impact** | Low — the test implementer will figure it out, but the ticket should be explicit to avoid ambiguity. |
| **Suggested fix** | Specify: "Check that `txn.requirements` contains `fastapi>=0.115` and that staged files include `app/main.py`, `app/__init__.py`, and `requirements.txt` with `fastapi>=0.115` in its content." |

## 8. Final Verdict: NEEDS REVISION

2. **B2**: `test_orchestrator_rollback_on_failure` — Fix description to match actual behavior: `rollback()` deletes `.forge-staging`, it does NOT keep it "for diagnostics."

### Moderate Issues to Fix Before Implementation

3. **M1**: Add mock executor injection to `orchestrator` fixture to prevent real `uv add` subprocess calls from `PluginExecutionEngine`.
4. **M2**: Align AC-2 with fixtures (use `MockProgressReporter`, not `StdoutProgressReporter`).
5. **M3**: Reconcile `real_registry` fixture between conftest and `test_validation.py` (promote or rename).
6. **M4**: Fix `test_orchestrator_empty_project` description to reference `backend_id=""` (not "zero domains").
7. **M5**: Specify assertions for `test_fastapi_plugin_execution_engine_real_plugin` — clarify whether "deps" means `txn.requirements` or `requirements.txt`.

### Non-Blocking Recommendations

- N1-N6: Various tightening suggestions (cli_spec_json structure, test count, capsys/pytest.raises mentions, boundary explanation).
